## Context

**架構盤點結果**（決定放哪裡的依據）：

系統目前有三處「條件式清單」機制，各有定位：

| 位置 | 定位 | 篩選機制 | 適合放同名嗎 |
|---|---|---|---|
| 學員管理 `/students` 快捷視圖 | 日常查詢辨識，隨時套用/取消 | `StudentView` + `matchesPostFilter` | ✅ 用途相符 |
| 資料維護 `/maintenance` 分類 | 待修清單，補完就消失 | `MaintenanceCategory`（欄位為 null） | ⭕ 也可，但同名未必是錯誤、不會「補完消失」 |
| 儀表板清單 | 統計與預警 | 伺服端彙整 | ❌ 非清單操作場景 |

使用者確認用途為**日常查詢辨識**，故採**學員管理快捷視圖**。

**關鍵技術現況**（`lib/db/supabaseRepository.ts`）：篩選有兩條執行路徑——
- 無跨欄位條件 → SQL 下推 + `range()` 分頁（高效）
- 有 `courseStage`/會籍/新生/`view` → **全量載入（每 1000 筆一批拉滿）+ JS 過濾 + slice 分頁**

既有 4 個快捷視圖都走第二條路徑，且**已經全量載入**。「同名」需要全體姓名統計才能判定，正好與此路徑相容——**不需引入新的查詢機制或 DB 變更**。

## Goals / Non-Goals

**Goals:**
- 一鍵列出當前體系內姓名完全相同（≥2 人）的學員，同名者相鄰顯示以利比對。
- 完全沿用既有快捷視圖的互動、體系隔離與「匯出＝畫面所見」。
- 不新增 DB 欄位、不新增 migration。

**Non-Goals:**
- **不做模糊比對**（同音字、簡繁、去除所有空白後比對）——使用者選擇「姓名完全相同」，避免誤判。
- **不做自動合併/去重**——本功能只負責「找出來看」，是否為同一人、要不要處理由人判斷。
- 不跨體系比對同名。
- 不在資料維護專區另開分類（本次僅快捷視圖）。

## Decisions

### D1. 放在學員管理的快捷視圖（而非資料維護分類）
使用者用途是「日常查詢辨識」，需要隨時套用/取消、與其他篩選並用；維護專區的語意是「待修正、補完即消失」，但同名學員**可能兩位都是真實存在的不同人**，永遠不會「消失」，放那裡會變成永遠清不掉的待辦。
- **替代**：維護專區新增分類——語意不符，且使用者已明確選擇日常查詢用途。若日後要做資料清理專用視角，可另開 change。

### D2. 判定規則：`name.trim()` 相等且出現次數 ≥ 2
只去除**前後**空白（`trim`），不去除中間空白——「王 小明」與「王小明」視為不同，符合「姓名完全相同」的選擇。空字串（`trim` 後為空）一律排除，避免大量無名資料互相判定為同名。

### D3. 統計時機：在 `runPaged` 全量載入後、per-row 過濾前
`matchesPostFilter(s, filters, now)` 是逐列判斷，無法得知全體統計。故：
1. 全量載入後，若 `filters.view === 'duplicate_name'`，先掃一遍建立 `Map<trimmedName, count>`。
2. 把「重複姓名集合」（count ≥ 2）以參數傳入過濾階段判定。
3. 過濾後、slice 分頁前，依姓名分群排序（同名相鄰），同名內再依 id 排序以求穩定。

實作上讓 `matchesPostFilter` 多接一個選用參數（重複姓名集合），或在 `runPaged` 內就地過濾——採前者以維持既有結構一致。

> 效能：全量載入本來就會發生（既有視圖已是如此），額外成本僅一次 O(n) 統計與一次排序，資料量級（數千筆）可忽略。

### D4. 匯出同步支援
`app/api/export/route.ts` 自行實作了一份快捷視圖過濾（與 repository 平行的邏輯）。為維持「匯出＝畫面所見」，需在該處一併加入同名判定（同樣先統計再過濾）。
- **注意**：這是既有的**邏輯重複**（repository 與 export 各一份）。本次沿用現狀、兩邊都加，不順手重構以控制風險；若日後要收斂，應另開 change 抽共用模組。

### D5. UI：第 5 個快捷視圖按鈕
`FilterBar.tsx` 既有 4 個快捷鈕（互斥），新增「同名學員」為第 5 個，沿用 `toggleQuickView` 既有互斥邏輯，無需改 store。

## Risks / Trade-offs

- [同名不等於重複資料] → 手冊/UI 措辭用「同名學員」而非「重複資料」，避免使用者誤以為要刪除其一。本功能只做辨識。
- [全量載入成本] → 既有視圖已全量載入，未新增額外往返；僅多 O(n) 統計。若日後資料量成長到需優化，應整體改為 SQL 端彙整（可另開 change），非本次範圍。
- [export 與 repository 邏輯重複] → 兩處都要改，漏改會導致「畫面有、匯出沒有」。驗證清單明列此項交叉檢查。
- [中間空白差異被視為不同名] → 為「完全相同」規則的直接後果；若實務上發現漏網（如「王 小明」），可再調整規則，屆時只需改判定函式一處。

## Migration Plan

無 DB 變更、無資料遷移。部署後即生效；舊有網址若帶 `view` 參數不受影響（新值為新增，不影響既有四種）。

回滾：移除該視圖選項即可，無殘留狀態。
